## Reasoning Flow Report

### Phase 1: CBR - Pattern Discovery

**[Engineer:Phase1:CBR]**

#### Similar Implementations and Patterns

**Rust Ownership System (Linear Types)**
- The λ◦ multiplicity system mirrors Rust's ownership model: `0` (erased/unused), `1` (linear/owned), `ω` (shared/borrowed), `&` (borrow observer)
- Key difference: Rust enforces at runtime with `Drop`, λ◦ enforces at compile-time via type system
- Pattern adapted: Context splitting (`Γ₁ + q·Γ₂`) corresponds to Rust's borrow checker rules for moving vs. copying values

**Linear Haskell (Quantitative Type Theory)**
- Atkey's quantitative type theory provides the formal foundation for the {0, 1, ω} semiring
- λ◦ extends this with the mode axis (`&`), distinguishing observation from quantification
- Pattern adapted: The `scale` operation on contexts mirrors Haskell's usage cardinality

**Idris 2 (Linear Types, Totality)**
- Idris 2's linearities and totality checking influenced the design
- λ◦ differs by using interaction nets instead of a CEK machine or Krivine machine
- Pattern adapted: Match/view distinction is similar to Idris's consumption checking

**OCaml (Modules, Traits)**
- One-file-one-module discipline parallels OCaml's compilation units
- Orphan rule implementation is identical to Haskell's coherence rules
- Trait resolution via DAG-DFS is similar to resolution in Scala implicit

**Lafont Interaction Nets (Original Formalism)**
- Original 1990 paper defines the agent set {λ, @, δ, ε}
- λ◦ extends with {Prim, PrimVal, PrimIO, IOToken} for practical computation
- Pattern adapted: The uniform evaluator loop (v2.4) eliminates the hybrid evaluator gap

#### audit-code Checklist Applied

- Function sizes are appropriate (most under 50 lines, complex functions documented)

- Parsimony: The net/step() dispatch is minimal and elegant

**Architecture** ✅
- Clear separation: `core/` (type system), `runtime/` (evaluator), `traits/`, `modules/`
- Each layer has distinct abstraction: AST → Type-checked AST → Net → Evaluated Result
- Explicit dependencies: no hidden coupling between modules
- Deep modules: `net/mod.rs` provides rich interaction model with 8 interaction rules

**Type Safety** ✅
- No `any` or `unknown` types used
- Generics used correctly in `Context<T>`, `Registry<K,V>`, `HashMap` operations
- Type-safe boundaries: `NodeId`, `PortIndex`, `WireId` as newtypes prevent mixing

**Correctness** ✅

**Properties** ✅

**Security & Performance** ⚠️

### Phase 3: TRIZ - Contradiction Resolution

**[Engineer:Phase3:TRIZ]**

#### Contradiction 1: Expressiveness vs. Safety

**Physical Contradiction**: 
- Expressiveness requires: user-defined data types, pattern matching, IO effects
- Safety requires: no runtime panics, no use-after-free, no data races

**Technical Contradiction**:
- Expressiveness: `ω` values should be duplicable for shared access
- Safety: duplication of mutable references causes data races

**Resolution (Separation Principle)**:
- The design separates quantity {0, 1, ω} from mode `&`
- `ω` implies immutability is stated as invariant (not mechanically enforced - design gap)
- Borrow mode `&` provides safe observation without duplication
- Match transfers ownership (_1), View provides observation (_&)

**Inventive Principle Applied**: 
- Principle 5: Merging - Prim/PrimVal agents merge primitive computation into uniform net evaluation loop (v2.4 resolves hybrid evaluator)
- Principle 10: Preliminary action - S5′ verification runs before parallel evaluation

#### Contradiction 2: Optimal Evaluation vs. Complexity

**Physical Contradiction**:
- Optimal evaluation requires: eager evaluation for primitives, lazy for control flow
- Complexity constraint: single uniform evaluator is easier to reason about

**Technical Contradiction**:
- Hybrid evaluator (v2.3): separate modes for lambda calculus vs. primitives
- Uniform evaluator (v2.4): Prim/PrimVal interactions fire identically to λ/@

**Resolution**:
- v2.4 eliminates hybrid evaluator by treating Prim as net agent
- All interactions fire via the same "find active pair, fire rule" mechanism
- Cost: Prim evaluation now requires scanning all nodes (no special-case fast path)

**Inventive Principle Applied**:
- Principle 12: Remove "one hand"
- The uniform evaluator loop removes the distinction between "net-reducible" and "primitive" computation

#### Contradiction 3: Separate Compilation vs. Coherence

**Physical Contradiction**:
- Separate compilation requires: modules compiled independently
- Coherence requires: at most one impl per (trait, type) pair globally

**Technical Contradiction**:
- Per-module checking (Gate 3): local coherence
- Link-time checking (L2): global coherence

**Resolution**:
- Orphan rule: impl legal only in module defining trait or type
- This makes coherence decidable per-module without whole-program analysis
- Two-phase checking: Gate 3 (local) + L2 (global)

**Inventive Principle Applied**:
- Principle 24: Intermediary
- The orphan rule acts as intermediary: modules can self-verify coherence before link
- Principle 25: Self-service
- Modules that pass orphan check don't need global verification for coherence

#### Contradiction 4: Observable Match/View vs. Implementation

**Physical Contradiction**:
- Observable difference: match transfers ownership, view only observes
- Implementation: both translate to similar net structures (ε for wildcards, direct wiring for vars)

**Technical Contradiction**:
- Match bindings receive `_1` multiplicity
- View bindings receive `_&` multiplicity
- But both use the same pattern translation mechanism

**Resolution**:
- Match: `translate_pattern` with `Multiplicity::One` → direct wire connections
- View: `translate_pattern` with `Multiplicity::Borrow` → same pattern nodes, different multiplicity annotation
- The operational difference is in ε-firing sites and borrow checker scope tracking

**Inventive Principle Applied**:
- Principle 1: Segmentation
- The type system segments bindings by multiplicity; the net translation respects this
- The borrow checker enforces the semantic difference at runtime

Citation: `[Engineer:Phase3:TRIZ] Four technical contradictions identified and resolved: (1) Expressiveness vs Safety via multiplicity/mode separation and ω-immutability invariant; (2) Optimal vs Simple evaluation via uniform evaluator loop (v2.4); (3) Separate compilation vs Coherence via orphan rule and two-phase checking; (4) Match/View observability via multiplicity-annotated pattern translation.`

### Synthesis

**[Engineer:Synthesis]**

The λ◦ implementation demonstrates a solid foundation in type theory and interaction net semantics, with the v2.4 design document providing comprehensive specification. However, two critical gaps prevent the implementation from matching its specification:

1. **IO System Incomplete** (HALT): `try_io_eval()` is specified but not implemented. The design's monadic IO system cannot execute IO actions.

2. **Constructor Patterns Dead Code** (HALT): The type checker handles `Pattern::Constructor` but the parser never produces it.

The core type system, multiplicity semiring, and interaction net evaluator are correctly implemented and verified by Lean4 proofs. The S5′ verification, coherence checking, and orphan rule are solid.

## Pre-mortem Analysis

**[Engineer:PreMortem]**

### Top 5 Failure Modes (ordered: likelihood × impact)

1. **IO System Never Implemented → Programs Cannot Perform IO**
   - **Root cause**: `try_io_eval()` was never implemented despite `PrimIO`/`IOToken` being defined. The design assumed it would be built but it wasn't prioritized.
   - **Signal**: Any user program with `Std.IO.println` or file operations hangs at evaluation. `net.step()` returns `InteractionResult::None` when only `PrimIO`/`IOToken` remain.
   - **Mitigation**: Implement `try_io_eval()` matching the PrimEval pattern. Create integration test with IO action that verifies completion.
   - **Likelihood**: High (gap has existed since v2.4 design)
   - **Impact**: Critical (IO is a major v2.4 feature)

2. **Constructor Patterns Remain Unimplemented → Exhaustiveness Checking Broken**
   - **Root cause**: Phase 5a was never started. Parser cannot produce `Pattern::Constructor`, but type checker has code paths for it.
   - **Signal**: `extend_with_pattern` with `Pattern::Constructor` is never called. User-defined algebraic data types cannot be pattern-matched.
   - **Mitigation**: Implement Phase 5a grammar extensions (type application, constructor patterns). Add compile-time check that dead code paths are removed or flagged.
   - **Likelihood**: Medium (Phase 5a is planned but not started)
   - **Impact**: High (blocks user-defined ADTs)

3. **S5′ Treats All ω as Mutable → Parallelism Under-utilized**
   - **Root cause**: No mechanism to distinguish immutable shared data from potentially mutable shared data. S5′ uses conservative structural check.
   - **Signal**: Parallel evaluator rejects programs where δ-agents share root δ-agents even if the shared data is immutable (e.g., shared constant lookup table).
   - **Mitigation**: Add `Immutable` type-level marker for ω values. Update S5′ to check mutability, not just structure.
   - **Likelihood**: High (design acknowledges this limitation)
   - **Impact**: Medium (programs are safe but not as parallel as possible)

4. **Orphan Rule Gap at Gate 3 → Conflicting Impls Discovered Late**
   - **Root cause**: Orphan rule is enforced per-module at Gate 3, but global coherence check at L2 is the real guarantee. A module might pass Gate 3 with a conflicting impl that only fails at L2.
   - **Signal**: `CoherenceViolation` error at link time when two modules define impl for same (trait, type) pair. Error message lists both modules.
   - **Mitigation**: Improve error messages at Gate 3 to indicate "potential conflict at L2". Consider warning instead of error.
   - **Likelihood**: Low (orphan rule is well-understood)
   - **Impact**: Medium (link-time error is confusing to users)

5. **ω-implies-immutability Invariant Violated → Data Races in Parallel Evaluation**
   - **Root cause**: No mechanical enforcement that ω values are immutable. A buggy compiler transformation could mutate shared data.
   - **Signal**: S5′ verification passes but parallel execution produces incorrect results. Race condition observable in traces.
   - **Mitigation**: Document as compiler-internal invariant. Add runtime assertions in borrow checker. Consider type-level proof.
   - **Likelihood**: Low (invariant is documented and respected by current implementation)
   - **Impact**: Critical (breaks memory safety guarantee)

### Systemic Risks

1. **Design-Implementation Drift**: The v2.4 design document is comprehensive but the implementation has significant gaps. This suggests the design process and implementation tracking are not synchronized.

2. **Dead Code Accumulation**: Constructor pattern handling in type checker is dead code. This pattern could continue - future features might be "specified but not implemented" without clear tracking.

3. **Missing Verification Gate**: There's no automated check that "all specified agents have handlers" or "all designed features are implemented". Phase completion is declared based on design document status, not implementation completeness.

4. **IO Monad Assumption**: The monadic IO model assumes `try_io_eval()` will be implemented. If this never happens, the entire IO system is vaporware.

### Pre-mortem Synthesis

**What class of problems dominates?**
The implementation suffers from **incomplete feature syndrome** - features are designed in detail but not implemented before the project claims "phase complete". The IO system is the most critical example, but constructor patterns and module system have the same pattern.

**What design assumption is most fragile?**
The assumption that "the design document is authoritative for what exists" is most fragile. The design specifies `try_io_eval()` as part of the evaluator, but the implementation doesn't have it. Anyone reading the design would expect IO to work.

**What does monitoring not cover?**
- No monitoring for "specified but not implemented" features
- No runtime metric for "did IO action execute or did evaluation get stuck?"
- No test coverage for IO system (since it doesn't exist)

Citation: `[Engineer:PreMortem] Top 5 failure modes: (1) IO system never implemented - critical impact, high likelihood; (2) Constructor patterns unimplemented - high impact, medium likelihood; (3) S5′ conservative - medium impact, high likelihood; (4) Orphan rule late detection - medium impact, low likelihood; (5) ω-immutability violated - critical impact, low likelihood. Systemic risk: design-implementation drift.`

## Concerns

### HALT CANDIDATE: IO System Incomplete

**Location**: `runtime/net/mod.rs` - missing `try_io_eval()` method

**Problem**: 
- `PrimIO` and `IOToken` agents are defined in `node.rs:47-54`
- The evaluator's `step()` function at lines 158-183 does not attempt IO evaluation
- User code cannot execute IO actions - `Std.IO.println`, file operations, etc. cannot work

**Feasibility Issue**: This is an implementation gap, not an impossibility. The IO operations can be implemented using Rust's standard IO primitives.

**Recommendation**: Implement `try_io_eval()` in `net/mod.rs` that:
1. Finds `PrimIO` nodes with connected `IOToken` and arguments
2. Executes the corresponding Rust IO operation
3. Produces result `PrimVal` and fresh `IOToken`

### HALT CANDIDATE: Constructor Patterns Dead Code

**Location**: `core/typecheck/rules.rs:197-203` and `core/borrow/checker.rs:175-180`

**Problem**:
- `extend_with_pattern` handles `Pattern::Constructor` 
- Parser (`grammar.rs:718-731`) only produces `Pattern::Wildcard` and `Pattern::Var`
- These code paths are never executed

**Feasibility Issue**: This is dead code that should either be removed or implemented alongside parser support.

**Recommendation**: Either:
1. Remove constructor pattern handling from type checker until parser supports it
2. Implement parser support for constructor patterns in Phase 5a

### Concern: ω-implies-immutability Not Enforced

**Location**: Design invariant stated in §14, not mechanically enforced

**Problem**: The design states "ω implies immutability" but no type-level or runtime mechanism prevents mutation of ω values.

**Feasibility Issue**: This is a design limitation, not an implementation gap. The fix would require type-level tracking of mutability.

**Recommendation**: Document as compiler-internal invariant. Consider type-level `Immutable` marker for future versions.

### Concern: Module System Not Implemented

**Location**: Entire `modules/` directory is scaffold, not implementation

**Problem**: 
- Design document v2.4 §11 fully specifies module system
- Implementation has `Module`, `Exports`, `Serializer` types but no actual compilation pipeline
